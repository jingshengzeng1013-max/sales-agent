
# 项目架构说明

## 1. 项目定位

该项目是一个面向中国政府采购网的本地数据采集与结构化处理系统，当前主链路是：

`列表页爬取 -> 详情页补全 -> 结构化抽取 -> 本地 SQLite 查询/后续 RAG 预留`

项目以 Python 脚本为主，没有 Web 服务层，也没有正式的包管理或 ORM 层；各模块通过共享配置和 SQLite 数据库协作。

## 3. 分层说明

### 3.1 `src/config.py`

- 统一管理项目根目录、数据目录、HTML 目录、数据库路径。
- 维护两个核心配置：
  - `CRAWLER_CONFIG`：列表页抓取参数。
  - `DASHSCOPE_CONFIG`：LLM 抽取参数，走 OpenAI 兼容接口。
- 启动时自动确保 `data/` 和 `html/` 目录存在。

### 3.2 `src/crawler/`

负责原始数据采集，分成两段：

- `ccgp_crawler.py`
  - 负责列表页抓取。
  - 从中国政府采购网搜索结果页解析项目标题和详情链接。
  - 将结果写入原始表 `tenders`。
  - 当前更偏“轻量入库”，先落标题和链接，再由详情流程补全字段。

- `crawl_detail.py`
  - 负责详情页抓取与清洗。
  - 从 `tenders` 中挑选待处理记录，抓详情页 HTML。
  - 解析正文、采购人、代理机构、预算、附件链接等信息。
  - 更新回 `tenders`，并把状态设为 `processed`。
  - 当正文异常或为空时，会把原始 HTML 保存到 `html/` 目录，便于后续修复。
  - 还包含 `export-bad-html`、`repair-content`、`repair-attachments` 等修复型命令。

### 3.3 `src/etl/`

负责从原始招标数据中抽取结构化字段。

- `extract_structured.py`
  - 读取 `tenders`。
  - 先做规则抽取，如公告类型、预算金额、省市、联系电话等。
  - 可选调用 DashScope/Qwen 做 LLM 增强抽取。
  - 将结果写入 `tender_structured`。
  - 支持初始化结构化表。

- `schema.sql`
  - 定义 `tender_structured` 和 `tender_chunks` 表。
  - 是当前 ETL 初始化数据库时使用的实际 SQL 来源之一。

- `query_tenders.py`
  - 交互式查询脚本。
  - 当前主要查询 `tenders` 原始表，还没有把 `tender_structured` 作为主要查询入口。

- `prompt_extract.md`
  - 保存结构化抽取 Prompt 文案。

- `llm_client.py`
  - 作为 LLM 调用封装层存在，但当前主抽取脚本中仍保留了部分内嵌调用逻辑。

### 3.4 `src/models.py`

- 当前承担“结构化字段单一真相源”的角色。
- 里面定义了：
  - 结构化字段枚举；
  - 默认 payload 模板；
  - `tender_structured` 建表 SQL 片段；
  - JSON Schema；
  - 轻量校验函数。
- 它更像“结构化设计稿 + 校验模型层”，但现阶段运行时主流程仍主要依赖 `src/etl/schema.sql` 与 `extract_structured.py`。

### 3.5 `src/utils/`

- `reset_db.py`
  - 删除并重建 SQLite 数据库。
  - 当前会重建 `tenders` 和 `crawl_logs`。

### 3.6 `tests/`

当前不是完整自动化测试体系，更偏向本地验证脚本：

- `verify_data.py`：查看总量、已处理数量、最近记录。
- `check_names.py`：检查项目名称、采购人等关键字段是否正常。
- `test_update.py`：验证数据库更新逻辑。

## 4. 核心数据流

### 4.1 阶段一：列表页入库

`src/crawler/ccgp_crawler.py`

- 构造搜索 URL。
- 请求列表页 HTML。
- 解析招标项目标题与详情链接。
- 写入 `tenders`。

这一阶段得到的是“原始索引层数据”，字段通常不完整。

### 4.2 阶段二：详情页补全

`src/crawler/crawl_detail.py`

- 查询 `status = 'new'` 或 `content` 为空的记录。
- 请求详情页。
- 提取正文、预算、采购人、代理机构、附件链接等。
- 回写到 `tenders`。

这一阶段把 `tenders` 从“列表索引表”提升为“原始事实表”。

### 4.3 阶段三：结构化抽取

`src/etl/extract_structured.py`

- 读取 `tenders` 中已有详情内容的数据。
- 执行规则抽取。
- 视参数决定是否调用 LLM。
- 写入 `tender_structured`。
- 同时保留 `extracted_json`，便于后续扩展字段或重放抽取结果。

### 4.4 阶段四：查询与后续能力

- 当前查询入口主要还是 `src/etl/query_tenders.py`。
- `tender_chunks` 已在 `schema.sql` 中预留，说明项目有继续走 RAG/检索增强的方向。
- 但目前还没有看到正式的分块写入流程或 API 服务层。

## 5. 数据库结构

### 5.1 `tenders`

原始招标事实表，主要字段包括：

- `project_name`
- `project_code`
- `buyer_name`
- `agency_name`
- `publish_date`
- `deadline`
- `budget`
- `region`
- `category`
- `detail_url`
- `content`
- `attachment_urls`
- `status`
- `created_at`

用途：

- 作为采集后的原始事实层。
- 保存网站原始字段，不应被结构化抽取结果覆盖。

### 5.2 `tender_structured`

结构化抽取结果表，当前用于承接规则抽取与 LLM 抽取后的结果。

典型字段包括：

- `announce_type`
- `buyer_name_std`
- `agency_name_std`
- `province`
- `city`
- `budget_raw`
- `budget_amount`
- `budget_unit`
- `product_keywords`
- `technical_requirements_summary`
- `contact_person`
- `contact_phone`
- `opportunity_score`
- `opportunity_reason`
- `next_action`
- `extracted_json`
- `llm_model`
- `llm_version`
- `extracted_at`

用途：

- 作为面向搜索、筛选、销售判断、后续 RAG 的结构化中间层。

### 5.3 `tender_chunks`

- 在 `src/etl/schema.sql` 中已经定义。
- 用于后续 RAG 分块与向量检索。
- 当前还处于预留阶段，未看到完整生产链路。

### 5.4 `crawl_logs`

- 由 `src/utils/reset_db.py` 创建。
- 用于记录爬取日志。
- 当前主爬虫流程中还没有看到统一写入该表的实现。

## 6. 模块之间的依赖方式

当前项目的依赖方式比较直接，特点如下：

- 各脚本通过 `sys.path.insert(...)` 把 `src` 加入导入路径。
- 各模块直接 `import config` 获取共享配置。
- 数据访问以 `sqlite3` 直连为主，没有 DAO、Repository 或 ORM 抽象层。
- 爬虫、ETL、查询、修复脚本均可单独执行，整体属于“脚本驱动型架构”。

## 7. 当前架构特点

### 优点

- 链路清晰，便于快速迭代。
- SQLite + 脚本的开发成本低，适合单机采集和验证。
- 原始表与结构化表已经初步分层。
- 已经为后续 RAG 留出 `tender_chunks` 扩展位。

### 现状限制

- 结构化定义存在双源：`src/models.py` 与 `src/etl/schema.sql` / `extract_structured.py` 之间还未完全统一。
- 查询层目前偏原始表导向，尚未充分利用 `tender_structured`。
- 还没有 API 层、任务调度层和统一日志层。
- `tests/` 更像验证脚本，不是稳定的自动化测试体系。

## 8. 推荐把它理解成什么架构

当前项目最适合被理解为：

**单机脚本驱动的采集与 ETL 管道**

可以进一步拆成四层：

1. 采集层：`src/crawler/`
2. 原始数据层：`tenders`
3. 结构化抽取层：`src/etl/` + `tender_structured`
4. 检索/RAG 预留层：`tender_chunks`

## 9. 现阶段最重要的几个入口

```bash
python src/crawler/ccgp_crawler.py
python src/crawler/crawl_detail.py
python src/etl/extract_structured.py --init-db
python src/etl/extract_structured.py --limit 50
python src/etl/query_tenders.py
python src/utils/reset_db.py
```

## 10. 一句话总结

这个项目当前已经从“简单爬虫”演进成了一个本地化的招投标数据采集与结构化处理管道，核心骨架已经具备，但结构化 schema、查询层和后续 RAG 层还处在继续收敛和扩展的阶段。
