# Agentic RAG 入门指南

> 让你从零理解 Agentic RAG 的概念、原理和实践

---

## 一、从普通问答开始

### 1.1 传统 RAG 是什么？

想象你有一个图书馆（你的招标数据库），你问图书管理员：

> "最近有哪些通信类的招标？"

图书管理员会：
1. **检索** - 在图书馆里找到所有通信相关的书
2. **返回** - 把找到的书给你

这就是 **RAG（Retrieval-Augmented Generation）**：
- **R**etrieval（检索）：找到相关信息
- **A**ugmented（增强）：把找到的信息加入提示
- **G**eneration（生成）：让 LLM 基于这些信息回答

```
用户提问
    ↓
RAG 系统检索相关数据
    ↓
把数据 + 问题一起发给 LLM
    ↓
LLM 生成回答
```

### 1.2 传统 RAG 的局限

但如果你问更复杂的问题：

> "帮我找出预算超过500万的通信类招标，按客户评分排序，并分析这些客户的历史中标率"

图书管理员会困惑：
- "排序是什么顺序？"
- "客户评分怎么看？"
- "中标率需要计算哪些数据？"
- "我应该去查哪些书架？"

传统 RAG 只能做 **一步检索**，无法处理这种需要 **多步思考和计算** 的问题。

## 二、什么是 Agentic RAG？

### 2.1 Agent（智能体）是什么？

继续用图书馆的例子：

**普通图书管理员**：只能找书，其他不会

**Agent（智能体）**：不仅能找书，还能：
- 帮你计算数据
- 跨多个数据库查询
- 调用外部工具（发邮件、查天气）
- 自己规划下一步该做什么
- 做错了会自我纠正

**Agent = 有一个"大脑"知道什么时候该用什么工具**

### 2.2 Agentic RAG = RAG + Agent

```
传统 RAG：    检索 → 生成（一步到位）

Agentic RAG： 思考 → 规划 → 检索 → 验证 → 再检索 → ... → 生成
              ↑     ↑
           Agent 在这里指挥整个流程
```

**Agentic RAG 的核心变化**：

| 传统 RAG | Agentic RAG |
|---------|-------------|
| 一次检索 | 迭代检索（不够就继续查） |
| 直接返回 | 评估结果，不够就补充 |
| 单轮问答 | 多轮对话、复杂推理 |
| 被动回答 | 主动规划下一步 |
| 只读 | 可执行操作 |

## 三、Agentic RAG 的核心能力

### 3.1 能力一：自动规划（Planning）

**问题**：用户说"分析这个项目"

**Agent 思考**：
```
用户说要"分析"，但分析什么？
需要明确：
1. 是分析客户？还是竞争对手？还是项目本身？
2. 分析需要哪些数据？
3. 先查什么？后查什么？

决定：
- 先查项目基本信息（预算、技术要求）
- 再查客户画像（历史行为）
- 最后查竞争态势（历史中标单位）
```

### 3.2 能力二：迭代检索（Iterative Retrieval）

**场景**：用户问"这个客户采购过哪些相关产品"

**Agent 行为**：
```
第1次检索：用"XX公司 采购" → 找到3条结果

评估：结果太少，可能关键词不够准确

第2次检索：用"XX公司 招标 中标" → 找到8条结果

评估：找到了更多信息，但不确定是否完整

第3次检索：用"XX公司 2024 2025" → 找到12条结果

评估：结果稳定，够了
```

### 3.3 能力三：工具调用（Tool Use）

Agent 不仅能检索，还能做很多事：

```python
# Agent 的工具箱示例
tools = {
    "search_tenders": "搜索招标信息",
    "get_customer_profile": "获取客户画像",
    "send_email": "发送邮件",
    "create_task": "创建任务",
    "calculate": "数学计算",
    "fetch_weather": "查天气",  # 外部工具
}
```

**例子**：
```
用户："把这个项目信息发给张总"

Agent：
1. 调用 search_tenders 找到项目信息
2. 调用 get_customer_profile 找到张总邮箱
3. 调用 send_email 发送邮件
4. 返回"已发送"
```

### 3.4 能力四：自我纠错（Self-Correction）

**问题**：用户问"这个客户的平均中标金额是多少"

**Agent 第一次尝试**：
```
1. 检索客户历史招标 → 找到10条
2. 计算平均：(所有金额相加)/10 = 250万
3. 回答：平均250万
```

**发现错误**：
```
等等，有些项目是联合中标，金额需要除以中标单位数量
重新计算：(所有金额)/10 = 250万  ✓ 正确
```

## 四、与现有系统的对比

### 4.1 传统方式（你现在的系统）

```
用户点击"销售建议"按钮
    ↓
系统调用 LLM 生成建议
    ↓
返回一段文字
```

**局限**：
- 只能生成文字，不能执行操作
- 如果数据不够，LLM 会"编造"答案
- 无法处理"先查A，再查B，根据B的结果查C"这类链式问题

### 4.2 Agentic RAG 方式

```
用户说："帮我分析这个项目，给出销售建议，如果机会评分>80就发邮件提醒我"

Agent 思考：
1. 先查项目详情
2. 再查客户画像
3. 计算机会评分
4. 如果评分>80，调用发送邮件工具
5. 返回分析结果+邮件发送状态
```

## 五、如何在你的项目中实现

### 5.1 简单版本：单 Agent 架构

```
┌─────────────────────────────────────────────────────────┐
│                    SalesAgent                            │
│  ┌─────────────────────────────────────────────────┐    │
│  │  think() - 分析问题，决定调用什么工具           │    │
│  └─────────────────────────────────────────────────┘    │
│                          ↓                                │
│  ┌─────────────────────────────────────────────────┐    │
│  │  tools: {search, profile, analyze, email}      │    │
│  └─────────────────────────────────────────────────┘    │
│                          ↓                                │
│  ┌─────────────────────────────────────────────────┐    │
│  │  act() - 执行工具调用                             │    │
│  └─────────────────────────────────────────────────┘    │
│                          ↓                                │
│  ┌─────────────────────────────────────────────────┐    │
│  │  generate() - 基于结果生成回答                   │    │
│  └─────────────────────────────────────────────────┘    │
└─────────────────────────────────────────────────────────┘
```

### 5.2 核心代码示例

```python
# src/agent/sales_agent.py

class SalesAgent:
    """销售场景 Agent"""

    def __init__(self):
        self.retriever = DualRetriever(data_type="tender")
        self.llm = OpenAI(...)
        
        # 工具注册
        self.tools = {
            "search_tenders": self.search_tenders,
            "get_customer_profile": self.get_customer_profile,
            "analyze_project": self.analyze_project,
            "generate_suggestions": self.generate_suggestions,
        }

    def process(self, user_query: str):
        """
        处理用户查询的主流程
        """
        # Step 1: 理解意图，制定计划
        plan = self.plan(user_query)
        
        # Step 2: 执行计划（可能多步）
        results = self.execute(plan)
        
        # Step 3: 验证结果
        if not self.is_sufficient(results):
            # 数据不够，补充检索
            results.extend(self.retrieve_more(plan))
        
        # Step 4: 生成回答
        return self.generate(user_query, results)

    def plan(self, query: str) -> dict:
        """
        分析问题，制定执行计划
        """
        prompt = f"""分析这个问题，决定需要哪些步骤：

问题：{query}

可选工具：
- search_tenders: 搜索招标
- get_customer_profile: 获取客户画像
- analyze_project: 分析项目详情
- generate_suggestions: 生成销售建议

请用 JSON 格式返回计划：
{{
  "steps": ["step1", "step2", ...],
  "reasoning": "为什么需要这些步骤"
}}
"""
        response = self.llm.chat(prompt)
        return json.loads(response)

    def execute(self, plan: dict) -> list:
        """执行计划"""
        results = []
        for step in plan["steps"]:
            tool = self.tools.get(step)
            if tool:
                results.append(tool())
        return results

    def search_tenders(self, query: str):
        """搜索招标"""
        return self.retriever.search(query, top_k=10)
```

## 八、常见问题

### Q: Agentic RAG 一定比普通 RAG 好吗？

**不是**。选择取决于场景：

| 场景 | 推荐方案 |
|------|----------|
| 简单问答 | 普通 RAG |
| 需要多步推理 | Agentic RAG |
| 需要执行操作 | Agentic RAG |
| 追求速度和简单 | 普通 RAG |
| 复杂业务场景 | Agentic RAG |

### Q: 实现 Agentic RAG 需要很多额外开发吗？

**是的**，但可以渐进式实现：

```
阶段1（简单）：在现有基础上加意图分类
阶段2（进阶）：加工具调用
阶段3（完整）：加迭代检索和自我纠错
```

### Q: 需要特殊的模型支持吗？

**不需要**。普通 GPT-4 或 Qwen 就能做 Agentic RAG，关键是 **prompt 设计** 和 **工具调用框架**。
