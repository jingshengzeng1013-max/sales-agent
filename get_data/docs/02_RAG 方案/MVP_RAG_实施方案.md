# 基于招投标数据的销售支持 MVP 实施方案

## 1. 目标

本方案面向销售部门，基于当前已采集的政府采购公告数据，构建一个能够进行：

- 内容理解
- 实体识别
- 结构化抽取
- 检索增强问答（RAG）
- 销售机会发现

的最小可用产品（MVP）。

MVP 的目标不是一步到位做成完整智能平台，而是优先解决以下问题：

1. 销售能快速发现与自身产品线相关的项目机会。
2. 销售能按客户、地区、时间、预算、关键词搜索项目。
3. 销售能看到每条公告的结构化摘要，而不是只看原始正文。
4. 销售能基于历史项目和相似项目进行知识检索与问答。

## 2. 当前数据基础

当前项目已经具备较好的原始数据基础，核心字段包括：

- `project_name`
- `project_code`
- `buyer_name`
- `agency_name`
- `publish_date`
- `deadline`
- `budget`
- `region`
- `category`
- `detail_url`
- `content`
- `attachment_urls`

其中：

- `content` 已经开始向正文纯文本方向清洗。
- `attachment_urls` 已能保存详情页和下载区域中出现的附件链接。
- 原始 HTML 可保存在 `html/` 目录中，便于后续补抓、修复、重解析。

这意味着当前数据可以直接作为“原始事实层”，支撑下一步结构化抽取和 RAG。

## 3. MVP 范围

MVP 建议只做四层能力：

1. 原始数据层
2. 结构化抽取层
3. 检索层
4. 销售使用层

不建议在 MVP 阶段做的内容：

- 复杂图谱系统
- 多 Agent 自动决策
- 全自动销售流程编排
- 复杂权限系统
- 大规模分布式向量检索

## 4. 总体架构

建议采用如下链路：

1. 爬虫采集项目列表和详情页
2. 保存原始字段到 `tenders`
3. 对正文和附件进行清洗、补充解析
4. 对每条项目调用规则抽取和大模型抽取
5. 抽取结果写入结构化表
6. 对原文、摘要、附件摘要进行切块和向量化
7. 前端提供筛选、检索、问答、机会判断能力

可以抽象为：

```text
爬虫 -> 原始表 tenders -> 抽取任务 -> 结构化表 -> 检索索引 -> 销售查询/问答
```

## 5. 数据分层设计

### 5.1 原始层

保留现有 `tenders` 表，作为唯一事实来源。

原则：

- 原始字段尽量不被大模型覆盖
- 后续任何结构化结果都单独存储
- 所有二次加工结果都可回溯到原始记录

### 5.2 结构化层

建议新增 `tender_structured` 表，每条公告对应一条结构化记录。

建议字段如下：

- `id`
- `tender_id`
- `announce_type`
- `buyer_name_std`
- `agency_name_std`
- `province`
- `city`
- `budget_raw`
- `budget_amount`
- `budget_unit`
- `product_keywords_json`
- `application_scenario`
- `technical_requirements_summary`
- `contact_person`
- `contact_phone`
- `attachment_summary`
- `opportunity_score`
- `opportunity_reason`
- `next_action`
- `extracted_json`
- `llm_model`
- `llm_version`
- `extracted_at`

说明：

- 标准化字段单独存列，方便 SQL 查询。
- `extracted_json` 用于保留完整输出，方便以后扩展 schema。

### 5.3 检索层

建议新增：

- `tender_chunks`
- `tender_embeddings` 或向量索引

`tender_chunks` 每条可包含：

- `chunk_id`
- `tender_id`
- `chunk_type`
- `chunk_text`
- `chunk_order`
- `metadata_json`

`chunk_type` 可取值：

- `title`
- `content_summary`
- `requirement_chunk`
- `contact_chunk`
- `attachment_summary`

## 6. 先做哪些结构化字段

MVP 第一版建议只抽高价值字段，不求全。

最推荐的第一批字段：

1. `announce_type`
2. `buyer_name_std`
3. `province`
4. `city`
5. `budget_amount`
6. `product_keywords`
7. `technical_requirements_summary`
8. `opportunity_score`
9. `opportunity_reason`
10. `next_action`

这批字段已经足够支撑销售筛选与初步跟进。

## 7. 抽取策略

建议采用“规则抽取 + 大模型抽取”的双阶段方案。

### 7.1 规则抽取

适合由规则或正则完成的字段：

- 公告类型
- 日期
- 联系电话
- 预算金额原文
- URL
- 附件数量
- 部分采购人/代理机构字段

优点：

- 成本低
- 稳定性高
- 可解释性强

### 7.2 大模型抽取

适合由大模型完成的字段：

- 产品关键词
- 应用场景
- 技术要求摘要
- 销售机会评分
- 机会原因
- 下一步建议动作

优点：

- 能理解语义
- 能从长文本中抽象总结
- 更适合销售视角分析

## 8. 大模型抽取输入输出设计

### 8.1 输入内容

建议传给模型的内容包括：

- `project_name`
- `buyer_name`
- `agency_name`
- `budget`
- `publish_date`
- `content`
- `attachment_summary`（若已下载解析）

### 8.2 输出格式

必须要求模型输出严格 JSON。

参考 schema：

```json
{
  "announce_type": "",
  "buyer_name_std": "",
  "agency_name_std": "",
  "province": "",
  "city": "",
  "budget_raw": "",
  "budget_amount": null,
  "budget_unit": "",
  "product_keywords": [],
  "application_scenario": "",
  "technical_requirements_summary": "",
  "contact_person": "",
  "contact_phone": "",
  "attachment_summary": "",
  "opportunity_score": 0,
  "opportunity_reason": "",
  "next_action": ""
}
```

### 8.3 Prompt 原则

Prompt 需明确约束：

- 只输出 JSON
- 未提及信息不要猜测
- 所有事实必须来源于输入文本
- `opportunity_score` 范围固定为 `0-100`
- 联系方式必须来自公告正文或附件文本

## 9. 附件处理策略

附件在销售视角下价值极高，很多关键参数、评分标准、品牌倾向都在附件里。

建议 MVP 对附件至少做到：

1. 保存附件 URL
2. 建立附件下载任务
3. 对 PDF/Word/Excel 做文本抽取
4. 为附件生成摘要
5. 将附件摘要并入结构化抽取和 RAG

建议优先支持：

- PDF
- DOC/DOCX
- XLS/XLSX

如果时间不够，MVP 第一版可以先只保存附件 URL，并在第二阶段补做附件文本解析。

## 10. RAG 设计

### 10.1 不建议直接全量向量化

仅靠向量检索容易出现：

- 命中不稳定
- 无法做预算/地区/时间过滤
- 销售问句难以精确响应

更建议做“混合检索”。

### 10.2 混合检索方案

推荐流程：

1. 先做结构化过滤
2. 再做关键词检索或全文检索
3. 再做向量召回
4. 最后由大模型生成最终回答

### 10.3 检索时的元数据过滤

推荐支持以下过滤项：

- 时间范围
- 地区
- 采购人
- 代理机构
- 公告类型
- 预算区间
- 产品关键词
- 是否有附件

### 10.4 向量化对象

建议向量化这些内容：

- 项目标题
- 正文分块
- 技术要求摘要
- 附件摘要
- 销售摘要

### 10.5 切块方式

建议按语义分块，而不是机械按固定字数切。

可分为：

- 项目基本信息块
- 采购需求块
- 技术参数块
- 联系方式块
- 附件摘要块

## 11. 面向销售的核心使用场景

MVP 建议先支撑下面 5 类场景：

### 11.1 新机会发现

示例问题：

- 最近一周有哪些和光芯片相关的项目？
- 哪些项目预算较高、且属于高校或科研院所？

### 11.2 客户画像

示例问题：

- 某高校最近两年采购过哪些相关设备？
- 某单位是否持续采购某类产品？

### 11.3 代理机构画像

示例问题：

- 哪些代理机构高频发布光电、芯片、测试系统相关项目？

### 11.4 技术需求检索

示例问题：

- 哪些项目提到了耦合效率、波长范围、探测器性能？

### 11.5 跟进建议

示例问题：

- 哪些项目值得优先跟进？
- 跟进理由是什么？
- 下一步应该联系谁？

## 12. MVP 页面建议

不需要一开始做复杂前端，建议先用轻量方案。

### 12.1 推荐实现方式

优先建议：

- `Streamlit`

原因：

- 开发快
- 易于内部试用
- 适合展示搜索、表格、详情和问答

### 12.2 页面结构

#### 页面一：项目列表页

支持筛选：

- 时间
- 地区
- 采购人
- 预算
- 产品关键词
- 公告类型

展示字段：

- 项目名称
- 采购人
- 地区
- 发布时间
- 预算
- 机会评分
- 产品标签
- 是否有附件

#### 页面二：项目详情页

展示内容：

- 原始字段
- 结构化字段
- 销售摘要
- 技术要求摘要
- 联系人信息
- 附件链接
- 原公告链接

#### 页面三：智能问答页

用户输入自然语言问题，如：

- 最近三个月西北地区高校和芯片相关的项目有哪些？
- 某采购单位采购过哪些测试系统？

系统返回：

- 结构化答案
- 命中的项目列表
- 相关原文证据

## 13. MVP 技术选型建议

### 13.1 后端

- Python
- FastAPI

### 13.2 存储

MVP 阶段可以继续使用：

- SQLite 保存原始和结构化结果

如数据量继续增长，再考虑：

- PostgreSQL
- `pgvector`

### 13.3 检索

MVP 推荐：

- 结构化检索：SQLite / SQL
- 关键词检索：SQLite FTS 或简单全文检索
- 语义检索：轻量 embedding + 本地向量索引

### 13.4 任务执行

建议拆成几个任务：

- 爬虫任务
- 正文修复任务
- 附件回填任务
- 结构化抽取任务
- 向量索引任务

## 14. 分阶段实施计划

### 第一阶段：打通结构化链路

目标：

- 建立结构化表
- 跑通单条抽取
- 批量处理历史数据

交付：

- `tender_structured`
- 规则抽取脚本
- LLM 抽取脚本

### 第二阶段：搭建销售搜索页

目标：

- 支持按字段过滤
- 展示结构化结果
- 能查看原文与附件链接

交付：

- Streamlit 搜索页
- 详情页

### 第三阶段：加入 RAG 问答

目标：

- 对结构化字段和文本块建立索引
- 支持自然语言查询

交付：

- 向量索引
- 混合检索服务
- 销售问答页

## 15. 成功标准

MVP 是否成功，建议看这些指标：

1. 销售能否在 30 秒内判断一条项目是否值得跟进。
2. 销售能否快速搜到相关历史项目。
3. 系统能否按采购人、地区、预算、关键词做有效过滤。
4. 结构化抽取结果是否足够稳定，能支持业务使用。
5. 销售是否愿意每天至少打开一次系统。

## 16. 推荐的第一批交付物

建议你下一步优先做这 4 项：

1. 新建 `tender_structured` 表
2. 定义固定 JSON schema
3. 跑第一版结构化抽取任务
4. 做一个简单的 Streamlit 搜索和详情页面

## 17. 风险与注意事项

### 17.1 数据质量

注意以下问题：

- 采购人名称不统一
- 预算格式不统一
- 附件可能缺失或无法解析
- 公告类型混杂
- 正文存在噪声或格式问题

### 17.2 模型稳定性

必须尽量避免：

- 输出不符合 JSON
- 凭空猜测字段
- 将总结性语言误当事实

### 17.3 业务落地

不要只做“智能问答”，一定要让销售能：

- 筛
- 搜
- 看摘要
- 找联系人
- 做跟进判断

## 18. 一句话建议

这套 MVP 最合理的落地顺序是：

先把“结构化抽取 + 销售搜索页”做出来，再上 RAG 问答。

原因是：

- 结构化能力是基础
- 销售的高频需求首先是筛选和发现
- RAG 更适合作为第二层增强能力，而不是第一层入口
