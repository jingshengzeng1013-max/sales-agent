# 销售情报系统：360° 客户画像数据规范 (V1.0)

## 1. 方案概述
本规范定义了销售情报系统中“客户画像”的核心数据结构。通过聚合历史招标、中标及产品匹配数据，系统为销售人员提供从“基础信息”到“竞争态势”的全方位洞察，旨在实现**精准获客**与**商机自动化挖掘**。

---

## 2. 核心维度定义

### 2.1 基础信息 (Basic Information)
> **业务目标**：识别客户身份，支撑地域营销与片区划分。

| 字段名 | 类型 | 说明 | 业务应用 |
| :--- | :--- | :--- | :--- |
| `customer_name` | String | 采购单位标准名称 | 唯一标识，用于跨系统（如 CRM）关联。 |
| `province` | String | 所属省份 | 销售大区划分依据。 |
| `city` | String | 所属城市 | 精准定位地市级业务机会。 |
| `industries` | Array | 行业标签集 | 识别客户赛道（如：气象、应急、公安、能源）。 |

### 2.2 需求画像 (Demand Profile)
> **业务目标**：深度挖掘客户技术偏好，寻找产品切入点。

| 字段名 | 类型 | 说明 | 业务应用 |
| :--- | :--- | :--- | :--- |
| `tech_keywords` | Array | 技术关键词聚合 | **技术DNA识别**。分析客户对卫星通信、FPGA、DSP等技术的依赖度。 |
| `application_scenarios` | Array | 业务应用场景 | **痛点切入**。识别客户是用于“灾害预警”还是“日常办公”。 |
| `technical_summaries` | Array | 技术要求摘要 | **规格参考**。记录客户对性能、交付周期的历史硬性要求。 |

### 2.3 价值画像 (Value Profile)
> **业务目标**：量化客户商业价值，优先级排序。

| 字段名 | 类型 | 说明 | 业务应用 |
| :--- | :--- | :--- | :--- |
| `tender_count` | Integer | 历史招标总次数 | **活跃度指标**。反映客户采购需求的频率。 |
| `total_budget` | Float | 历史累计预算金额 | **购买力指标**。评估客户的资金实力。 |
| `avg_opportunity_score`| Float | 商机平均分 | **意向度指标**。评估客户与我司产品的整体契合度。 |

### 2.4 竞争属性 (Competitive Landscape)
> **业务目标**：知己知彼，制定存量替换或差异化竞争策略。

| 字段名 | 类型 | 说明 | 业务应用 |
| :--- | :--- | :--- | :--- |
| `past_winners` | Array | 历史中标供应商 | **竞品分析**。识别该客户的“朋友圈”和主要对手。 |
| `past_winning_products`| Array | 历史中标产品 | **替代机会**。分析竞品弱点，寻找我司产品替换空间。 |

### 2.5 历史足迹 (History Tenders)
> **业务目标**：追溯画像生成的数据依据，了解单体项目细节。

| 字段名 | 类型 | 说明 | 业务应用 |
| :--- | :--- | :--- | :--- |
| `project_name` | String | 历史项目名称 | 追溯具体业务背景。 |
| `budget` | Float | 单笔项目预算 | 了解单体项目规模。 |
| `winning_amount` | Float | 中标金额 | 了解实际成交价格。 |
| `date` | String | 标讯发布日期 | 掌握客户采购的时间周期。 |
| `url` | String | 原始标讯链接 | 点击可回溯查看完整的招标公告原文。 |

## 4. 后续优化方向 (LLM 赋能)
1.  **自动行业归类**：利用 LLM 根据单位名称自动填充 `industries` 标签。
2.  **技术 DNA 总结**：将零散的技术要求总结为一段 50 字的“客户技术偏好描述”。
3.  **销售话术生成**：结合画像自动生成针对该客户的“首封开发信”或“电话开场白”。
